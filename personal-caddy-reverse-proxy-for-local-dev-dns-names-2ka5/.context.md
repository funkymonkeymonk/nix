# Goal

Replace the current one-shot `caddy reverse-proxy --from vane.localhost:80
--to 127.0.0.1:3000` with a **persistent personal Caddy service** that
reverse-proxies every local dev stack under distinct DNS names on port 80,
with its admin API enabled so any devenv can add/remove its own routes
dynamically via HTTP.

End state: every local service gets a clean URL like
`http://<service>.<stack>.localhost` — no port numbers anywhere in the
URLs the user types.

# Background

Today `ps` shows:

```
caddy reverse-proxy --from vane.localhost:80 --to 127.0.0.1:3000
```

This is a one-shot `reverse-proxy` subcommand that:
- Serves exactly one route
- Has no admin API (`curl localhost:2019/config/` returns empty)
- Can't be dynamically reconfigured
- Forces every other local service to use port-in-URL or take a
  different port-80 slot (conflict)

The motivating case: OKR cycle Temporal workflow (Justworks
devex-backlog repo) runs a docker-compose Temporal stack and wants its
UI reachable via `temporal.okr-cycle.localhost`. With only a one-shot
Caddy bound to port 80, it's stuck typing localhost:PORT — and has to
allocate ports dynamically to avoid colliding with every other local
service (Backstage, hiring test-harness, etc.).

A proper personal Caddy with:
- A real `Caddyfile` (or dynamic JSON config)
- The admin API enabled on `localhost:2019`
- launchd so it starts at login

…means every devenv can POST its routes to `localhost:2019/config/...`
on `enterShell` and DELETE them on exit. Routes are namespaced by
stack so conflicts are impossible. Users type
`http://temporal.okr-cycle.localhost`, `http://tilt.okr-cycle.localhost`,
`http://vane.localhost`, etc. — never a port.

      existing `modules/services/vane.nix`) manages a persistent Caddy
      service via `launchd.user.agents.caddy` on Darwin
- [ ] Caddy listens on `127.0.0.1:80`
- [ ] Caddy admin API listens on `127.0.0.1:2019`

      vane flow keeps working with no regression

      the current route set

# Notes

- macOS resolves `*.localhost` to 127.0.0.1 automatically. No /etc/hosts
  edits required.
- Binding to port 80 requires `SecureInput` etc. on macOS — Caddy via
  launchd with `<key>SoftResourceLimits</key>` handles privileged ports
  without root via the setuid mechanism (or via pfctl redirect from 80
  to a high port). Check current nix-darwin idioms.
- Admin API is enabled by default on port 2019 unless config explicitly
  disables it. Just don't include `"admin": {"disabled": true}` in the config.
- Dynamic route POSTs: `curl -X POST localhost:2019/config/apps/http/servers/srv0/routes/0 -d '{...}'`
  inserts at index 0. Match by @id to DELETE on exit.
- Alternative technologies to consider before implementing:
  - `colima` / `orbstack` (you have colima; orbstack gives free
    `.orb.local` DNS). Could swap colima for orbstack instead of
    building this.
  - `traefik` instead of caddy — similar shape, different ergonomics.
  - `dnsmasq` + manual port assignments (doesn't solve the DNS-name goal on its own).
- Keep this yak scoped to the Caddy side. Downstream integrations
  (okr-cycle devenv, hiring test-harness devenv, etc.) are their own
  follow-up PRs in their respective repos.
