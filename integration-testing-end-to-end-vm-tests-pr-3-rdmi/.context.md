# Goal
Boot real NixOS VMs using pkgs.testers.nixosTest to verify full-stack behavior — not just option evaluation, but actual service startup, package presence, and file installation.

# Architecture Decision: mkVmTest helper (Approach B)

## The type-server question
type-server is the production "cattle" NixOS config. Using it verbatim in VM tests is NOT possible because:
- It sets `hardware.facter.reportPath = "/etc/nixos/facter.json"` — a host filesystem path
- pkgs.testers.nixosTest runs in the Nix sandbox — no host filesystem access
- This would cause sandbox build failures; --impure doesn't help inside nixosTest

## The solution: inline facter stub via Nix store path
Replace the host path with a pkgs.writeText store path:
```nix
hardware.facter.reportPath = pkgs.writeText "facter.json" ''
  {
    "version": 1,
    "hardware": {
      "system": { "uuid": "00000000-0000-0000-0000-000000000000" },
      "cpu": [{ "architecture": "x86_64", "vendor_name": "GenuineIntel", "features": ["vmx"] }],
      "memory": { "total_bytes": 8589934592 },
      "disks": [{ "path": "/dev/sda", "size_bytes": 53687091200 }],
      "network_interfaces": [{ "name": "eth0", "mac": "00:00:00:00:00:00" }],
      "pci": [], "scsi": []
    },
    "boot": { "efi_available": true },
    "filesystems": [{ "mountpoint": "/", "device": "/dev/sda1", "fs_type": "ext4" }],
    "virtualisation": "none"
  }
'';
```
This is pure (file in Nix store), sandbox-compatible, and lets us import the full ./modules tree.

## Two test helpers
- `mkTest` (existing, keep): stripped-down, only imports options.nix + roles/default.nix + nixos/base.nix. Use for basic behavior tests (users, SSH, packages).
- `mkVmTest` (new for Tier 4): imports full ./modules tree + home-manager + disko + inline facter stub. Use for full-stack role/service tests.

## What mkVmTest unlocks
- Full ./modules tree (all roles + services + home-manager modules)
- home-manager module activation (e.g., verify opencode config file generated at ~/.config/opencode/config.json)
- disko option evaluation (no actual partitioning in QEMU — that's fine)
- Service systemd unit startup (e.g., ollama.service)
- Skills SKILL.md file installation via home.file entries

## What mkVmTest does NOT test (acceptable)
- Actual nixos-facter hardware detection — install-time concern, not config correctness
- Real disk partitioning — disko's own test suite covers that
- 1Password/opnix secrets — legitimately cannot be tested in CI sandbox; disable with onepassword.enable = false
