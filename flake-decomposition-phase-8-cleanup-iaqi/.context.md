# Problem
Once all machines are migrated to hosts/+archetype pattern and the dust settles, wire flake-parts properly and use it to support self-updating instance generation. Superseded the original plan's premise (schema/profiles split) — see project/plans/2026-07-19-flake-decomposition-phase7-phase8-plan.md for full context on the pivot.

## Prerequisites
- Flake decomposition: Phase 7 — migrate wweaver must be done

# Background / pivot
Original design doc (Section 2-3) planned a 3-flake split (library/schema/profiles).
PR #337 deleted profiles/ and schema/ as unused. PR #348 replaced them with a
simpler base-darwin.nix + workstation-darwin.nix archetype pattern, composed
directly in flake.nix. wweaver/MegamanX/darwin-server already moved onto this
pattern directly (no -v2 twin, no closure-diff gate). flake-parts input is
declared but never wired into outputs.

New driver for flake-parts (confirmed 2026-07-19): generate a flake for a new
machine instance by composing multiple flake parts (library archetypes +
machine overrides), and have that instance keep itself up to date
automatically. This is the actual motivation for finishing the flake-parts
wiring — not composition aesthetics.

# Key constraint
Darwin machines cannot self-upgrade natively — nix-darwin's system.autoUpgrade
does not exist yet (PR #1682, still open as of 2026-07-19). Any Darwin
instance-generation prototype in 8.6 will still need manual/cron-triggered
darwin-rebuild switch until that upstream PR lands. Prototype the
self-updating story on NixOS first (zero/type-server already have working
autoUpgrade.flakeUrl).
