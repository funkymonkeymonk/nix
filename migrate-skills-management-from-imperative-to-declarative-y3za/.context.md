# Goal
Move internal + superpowers skill delivery for opencode and claude-code from
hand-rolled home.file entries (modules/home-manager/skills/install.nix,
modules/home-manager/claude-code.nix, modules/home-manager/pi-coding-agent.nix)
to home-manager's native programs.opencode.skills / programs.claude-code.skills
options, where the content is known at Nix eval time. This closes the
imperative-vs-declarative gap for the two source types that CAN be expressed
statically. External skills (installed via npx skills add at activation time)
cannot become declarative -- see Non-Goals.

## Prerequisites
- Deduplicate auto-loaded skills logic across home-manager modules must be done
  (build the native-options migration on top of one shared skills-loading
  helper, not three/four still-separate implementations)

# Background
Investigated 2026-07-19 while consolidating the pi/opencode/claude-code role
and home-manager layers. Findings:

- home-manager's native programs.opencode.skills and programs.claude-code.skills
  options already exist and accept exactly the shape this repo produces:
  attrsOf (string-or-path), or a path to a directory containing one folder
  per skill. Verified via nix eval against inputs.home-manager modules/
  programs/{opencode,claude-code}.nix -- confirmed both options exist with
  matching type signatures.
- pi-coding-agent has NO native home-manager module (there is no
  programs/pi-coding-agent.nix upstream) -- pi's skill delivery must stay
  a custom home.file-based implementation regardless of this migration.
- Today there is a real capability gap, not just a mechanism difference:
  skills/install.nix (used by opencode) installs EVERY enabled skill as
  its own directory under ~/.config/opencode/skills/<name>/, plus bundled
  commands. claude-code.nix and pi-coding-agent.nix only install the
  subset tagged autoLoad = true, concatenated into one digest file
  (~/.claude/CLAUDE.md / ~/.pi/agent/AGENTS.md). Manifest entries like
  "nix-adding-services" list roles = ["opencode" "claude" "pi"] implying
  parity across all three, but only opencode actually gets full skill dirs
  today. This gap should close as part of (or immediately after) the
  auto-load-dedupe prerequisite, using the "maximal choice" pattern (give
  claude/pi the same full-directory capability opencode already has) --
  confirmed with the user this is safe: pi is live on both real machines
  (wweaver, MegamanX) and the change is additive (new directories appear
  alongside existing content, nothing is removed); claude is dormant on
  every real machine today (roles.claude.enable is never set to true
  anywhere; wweaver explicitly sets claude-code.enable = false), so there is
  zero observable effect on it until/unless it's turned on somewhere.
- The npx skills add activation script (modules/home-manager/skills/install.nix,
  external source type) fetches skill content over the network AT ACTIVATION
  TIME, not eval time. It writes into the SAME target directories
  (~/.config/opencode/skills/, ~/.claude/skills/, ~/.pi/agent/skills/ -- see
  vercel-labs/skills agent path table) that the native options would manage
  for internal/superpowers skills. This tool is the open agent skills CLI
  (https://github.com/vercel-labs/skills, "npx skills"), already wired via
  the vercel-skills flake input (currently flake=false, used only as a
  reference/pinned commit -- the actual installer is invoked via bare npx
  --yes skills@latest, not from the pinned flake input content). No manifest
  entry currently uses source.type = "external" in practice (the only
  example in manifest.nix is commented out), so this code path is currently
  unexercised in production but the mechanism must be preserved for when it
  is used.

# Acceptance Criteria
- [ ] modules/home-manager/opencode.nix uses programs.opencode.skills for
      internal + superpowers skills, sourced from the shared skills-loading
      helper (built in the prerequisite yak), instead of hand-rolled home.file
      entries in skills/install.nix
- [ ] modules/home-manager/claude-code.nix uses programs.claude-code.skills for
      internal + superpowers skills (closing the full-skill-dir gap vs
      auto-load-only, per Background) via the same shared helper
- [ ] modules/home-manager/pi-coding-agent.nix's skill delivery is upgraded to
      the same full-directory capability (closing the auto-load-only gap) but
      necessarily stays home.file-based since there's no native
      programs.pi-coding-agent module upstream
- [ ] External skills (source.type = "external") continue to be installed via
      the npx skills add activation script, writing to the same final target
      directories as before -- this mechanism is explicitly OUT of scope for
      "declarative" conversion (see Non-Goals) but must keep working
      unmodified and merge correctly with the now-native-managed
      internal/superpowers skills (verify no directory-ownership conflict
      between home-manager-managed symlinks and activation-script-written files)
- [ ] Bundled commands (skill.commands, e.g. yak-shaving's /shave,
      jj's /finish /pr etc.) still install correctly under the new mechanism
- [ ] devenv tasks run check:lint and devenv tasks run test pass
- [ ] Manually diff the resulting ~/.config/opencode/skills/,
      ~/.claude/skills/, ~/.pi/agent/skills/ directory listings before/after
      on a real build (not just eval) to confirm no skill silently
      disappears or duplicates due to the mechanism swap

# Non-Goals
- External skill installation (npx skills add) does NOT become declarative --
  it is fundamentally a runtime network fetch, not expressible as a static
  Nix value. This yak preserves that activation-script mechanism as-is.
- Not migrating pi-coding-agent's OTHER config surfaces (settings.json,
  models.json, extensions, npmPackages, etc.) -- only skill delivery.
