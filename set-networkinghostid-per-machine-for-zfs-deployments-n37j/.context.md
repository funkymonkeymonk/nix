# Problem

`networking.hostId` is a ZFS requirement — it must be a unique 8-hex-digit value
per physical machine. ZFS uses it to prevent pool imports on the wrong host.

Surfaced in PR #283 (feat/type-nas): `targets/type-nas/default.nix` set
`networking.hostId = lib.mkDefault "8badf00d"` as a placeholder. This conflicted
with `networking.hostId = "71952ca1"` injected by `microvm.nixosModules.microvm`
(included in all NixOS cattle configs in `flake.nix`). Both resolve to the same
Nix priority, causing an eval error.

The placeholder was removed as a workaround, but the underlying problem remains:
there is no mechanism in this repo to set a unique `hostId` per ZFS deployment.

## What nixos-facter does NOT do

`hardware.facter.reportPath = "/etc/nixos/facter.json"` collects CPU, memory,
network interface hardware facts. It does NOT generate or report `networking.hostId`.
That value must be explicitly set — it is a NixOS concept, not a hardware fact.

## Current state

- `targets/type-nas/default.nix` has NO `networking.hostId` set (removed in PR #283 fix)
- The `microvm` module sets `"71952ca1"` as a side-effect on all NixOS configs
- Any type-nas deployment will silently use the microvm module's placeholder value
- Multiple type-nas machines would all share the same hostId — defeating ZFS's
  import guard and risking pool corruption if disks are moved between machines

## Options

### Option A: cloud-init / first-boot generation (recommended for cattle)
Add a systemd one-shot service that generates a unique hostId on first boot and
writes it to `/etc/nixos/hostid` (or similar), then have the NixOS config read
it. This fits the cattle model — no per-machine config in the flake.

### Option B: deploy script
Have `scripts/install-machine.sh` (or equivalent) generate a hostId and pass it
as a nixos-rebuild argument or write it to a well-known path before installation.

### Option C: document it as a manual step
Add to deployment docs: "run `head -c4 /dev/urandom | od -A none -t x4 | tr -d ' \n'`
and set `networking.hostId` in a host-specific overlay." Least code, most fragile.

## Acceptance Criteria

- [ ] Each type-nas (and any future ZFS-using) deployment gets a unique, stable hostId
- [ ] The mechanism works without per-machine entries in flake.nix (cattle model)
- [ ] The microvm module's injected hostId does not silently win on ZFS hosts
- [ ] Deployment docs describe how hostId is set/generated
- [ ] `devenv tasks run test:nixos-eval` passes with the solution in place

## Files

- `targets/type-nas/default.nix` — where ZFS config lives, needs hostId solution
- `flake.nix` — includes `microvm.nixosModules.microvm` in all NixOS configs (source of conflict)
- `machine-types/server.nix` — cattle server base, potential place for first-boot hook
- `scripts/install-machine.sh` (if it exists) — potential place for generation at install time
- `docs/` — deployment docs need updating regardless of chosen approach
